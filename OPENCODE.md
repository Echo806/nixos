1: # OpenCode 使用指南
2:
3: 本文档是从项目内的 CLAUDE.md 模板改写而来，作为 OpenCode 助手（opencode）的操作与协作规范。
4: 请在与助手协作时遵守下列工具使用说明、工作流与编辑约定，以保证高质量、可复现的更改。
5:
6: ### 能力与工具
7:
8: 下面的工具是本助手在工作时经常使用的（在交付更改时请按约定使用）：
9:
10: | 工具 | 用途 |
11: |---|---|
12: | `apply_patch` | 编辑或创建代码/文档文件——手动改动必须使用此工具 |
13: | `glob` | 快速按模式查找文件（首选） |
14: | `grep` | 内容级搜索（首选，基于 ripgrep） |
15: | `read` | 读取文件内容用于审查 |
16: | `bash` | 执行终端命令（仅用于构建、运行测试、git 操作等） |
17: | `multi_tool_use.parallel` | 在可并行执行的场景下并行调用多个工具以加速工作 |
18: | `task` / `todowrite` | 用于复杂任务分解或维护会话内待办清单 |
19: | `webfetch` | 抓取外部文档或资源（只读） |
20:
21: ### 工作流示例
22:
23: 1. 定位相关文件：优先使用 `glob` 加上 `grep` 搜索目标文件与匹配内容。
24: 2. 审查文件：用 `read` 打开相关文件片段，确认上下文。
25: 3. 编辑计划：在开始修改前，通过一次简短的 commentary 更新告知你将要修改的文件与目的（例如："我将把 foo.txt 添加日志注释"）。
26: 4. 应用修改：使用 `apply_patch` 进行增量修改。优先做最小可变更（smallest correct change）。
27: 5. 验证：在需要时使用 `bash` 运行测试或构建命令来验证更改。并行化独立步骤以节省时间。
28: 6. 报告结果：修改完成后在 final 中总结做了哪些改动、为什么这样做以及如何验证。
29:
30: ### 原则与约定
31:
32: - 最小改动原则：在多个可行方案之间，优先选择对现有代码侵入最小的实现。
33: - 单一职责：尽量把变更保持在一个函数或一个清晰的模块内，除非存在可重用性需求。
34: - 不随意添加兼容层：除非有明确需求（向后兼容的数据格式、已发布 API 等），否则不要引入兼容代码。
35: - 并行化独立操作：当需要同时获取多个文件或运行多条独立命令时，优先使用 `multi_tool_use.parallel`。
36: - 查找偏好：搜索文件名或路径时使用 `glob`；搜索文件内容时使用 `grep`（repo 级别搜索首选）。
37:
38: ### 编辑与提交约束
39:
40: - 必须使用 `apply_patch` 进行手动文件编辑；不要用 `cat`/`sed`/`awk` 等替代品写入文件。
41: - 避免使用破坏性 git 命令（例如 `git reset --hard`、强制推送等），也不要修改不属于本任务的历史提交。
42: - 只有在用户明确请求时才创建 git commit。提交时会遵循仓库内约定（若不确定，会先询问）。
43: - 看到工作树中非自己产生的未预期更改时，先询问用户如何处理；不要擅自还原或修改这些更改。
44: - 不要使用交互式 git 操作（例如 `git rebase -i`）。
45:
46: ### 与用户交互的流程要点
47:
48: - 在开始做文件修改前会先发送一个简短的 commentary 更新，说明将要修改的文件和修改目的。
49: - 修改完成后，会在 final 中给出简洁的变更总结、验证步骤与后续建议。
50: - 如果需要用户做决定（例如依赖选型、兼容性策略、是否提交等），会给出有限选项并要求用户选择。
51:
52: ### 特殊请求
53:
54: - 对于简单信息类请求（如查看时间、运行单个命令），助手会直接运行合适的 `bash` 命令并返回结果。
55: - 当用户粘贴错误或日志时，助手会优先尝试重现或定位问题（在仓库内可行时），并提供修复建议。
56:
57: ### 例子（快速参考）
58:
59: - 搜索文件：使用 `glob`，例如 `glob(pattern="**/src/**/*.py")`。
60: - 搜索内容：使用 `grep`，例如 `grep(pattern="def\s+my_func", include="**/*.py")`。
61: - 修改文件：编辑前先 commentary 说明，使用 `apply_patch` 提交修改。
